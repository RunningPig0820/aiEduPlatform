# 教育 AI 智能体平台技术方案（DDD 架构）

## 一、项目概述

本项目采用 **领域驱动设计 (DDD)** 构建后端核心，结合 **前后端一体化** 架构（Thymeleaf 服务端渲染），实现学生、老师、家长三端的生产可用功能。

------

## 二、三端功能需求清单

|   功能模块   |                           具体需求                           |

| **AI 备课**  | 1. 根据课题自动生成教案框架2. 推荐匹配的例题与作业题3. 支持在线编辑与保存教案 |
| **题库管理** | 1. 支持手动录入 / 批量导入题目2. 拍题识别录入（支持难认字校正）3. 题目知识点标签化管理 |

## 三、技术栈选型

### 3.1 后端核心技术（DDD 架构）

|     分类     |                           技术选型                           |                           选型理由                           |
| :----------: | :----------------------------------------------------------: | :----------------------------------------------------------: |
|   **框架**   | **Spring Boot 3.2+** + **Spring Modulith** (DDD 模块化支持)  | Spring Boot 构建稳健后端，Spring Modulith 提供 DDD 模块化验证与支持。 |
|   **ORM**    | **Spring Data JPA** (领域模型持久化) + **MyBatis-Plus** (复杂查询) | JPA 贴合 DDD 聚合根设计，MyBatis-Plus 处理复杂 SQL 与批量操作。 |
|  **数据库**  |  **MySQL 8.0** (InnoDB 引擎) + **Flyway** (数据库版本迁移)   | MySQL 事务支持好，Flyway 保证多人协作时数据库 schema 一致性。 |
|   **缓存**   |                 **Redis** (Redisson 客户端)                  |              缓存热点题库、用户会话、分布式锁。              |
| **权限认证** |   **Spring Security** + **Session** (Spring Session Redis)   |     实现 RBAC 权限控制，Session 集中存储支持多实例部署。     |
| **消息队列** |                  **RabbitMQ** (Spring AMQP)                  |     异步处理 AI 耗时任务（OCR、大模型推理），削峰填谷。      |
| **文件存储** |                   **MinIO** (兼容 S3 协议)                   |        开源对象存储，无需依赖云服务，适合私有化部署。        |
|  **工具库**  |     **Hutool** + **MapStruct** + **Lombok** + **Guava**      |                 减少样板代码，提升开发效率。                 |

### 3.2 前端技术（前后端一体化）

|     分类     |                       技术选型                        |                           选型理由                           |
| :----------: | :---------------------------------------------------: | :----------------------------------------------------------: |
| **模板引擎** |            **Thymeleaf** (Spring 官方推荐)            |       服务端渲染，SEO 友好，与 Spring Boot 无缝集成。        |
| **样式框架** | **Tailwind CSS** (v3) + **daisyUI** (Tailwind 组件库) |        快速构建现代美观界面，无需编写大量自定义 CSS。        |
| **交互脚本** |                  **Alpine.js** (v3)                   | 轻量级 “微型 Vue”，处理表单提交、模态框等简单交互，无需前后端分离。 |
|  **图表库**  |             **ECharts** (Apache ECharts)              |        生成学情趋势图、知识点雷达图等数据可视化图表。        |
|  **富文本**  |            **TinyMCE** (开源富文本编辑器)             |             老师备课、录入题目时支持富文本格式。             |

### 3.3 AI 服务技术（Python 微服务）

|     分类     |                           技术选型                           |                           选型理由                           |
| :----------: | :----------------------------------------------------------: | :----------------------------------------------------------: |
| **Web 框架** |                         **FastAPI**                          | 高性能异步框架，自动生成 API 文档，作为 AI 服务入口与消息队列消费者。 |
| **OCR 引擎** |                **PaddleOCR** (PP-StructureV2)                |       中文手写体识别准确率高，支持版面分析与表格识别。       |
|  **大模型**  | **通义千问 API** (Qwen2.5-Turbo) 或 **本地部署 Qwen2.5-7B-Instruct** |   云端 API 效果好，本地部署保护隐私，LangChain 统一编排。    |
| **AI 编排**  |                  **LangChain** (Python 版)                   |         管理 Prompt 模板、RAG 检索链、多轮对话记忆。         |
|  **向量库**  |             **Milvus** (Java/Python 双 SDK 支持)             |      存储题目与知识点向量，支持 “举一反三” 相似题检索。      |

### 4.2 DDD 分层说明

|               层级               |                           职责说明                           |
| :------------------------------: | :----------------------------------------------------------: |
|       **Domain（领域层）**       | 核心业务逻辑所在，包含实体、值对象、聚合根、领域服务、仓储接口，不依赖任何其他层。 |
|    **Application（应用层）**     | 编排领域服务，处理用例流转，负责 DTO 与聚合根的转换，不包含核心业务逻辑。 |
| **Infrastructure（基础设施层）** | 提供技术能力支持，实现仓储接口、数据库访问、消息队列、文件存储、AI 服务调用等。 |
|     **Interface（接口层）**      | 处理用户交互，包含 Thymeleaf 页面控制器（服务端渲染）和 REST API 控制器（异步交互）。 |

## 五、关键交互流程（DDD 事件驱动）

### 5.1 学生拍题上传流程

1. **接口层**：学生上传图片，`StudentController` 接收请求。
2. **应用层**：`HomeworkAppService` 编排流程，创建 `HomeworkAggregate`。
3. **领域层**：`HomeworkAggregate` 初始化，状态设为 `PROCESSING`，发布 `HomeworkSubmittedEvent`。
4. **应用层**：监听到事件，调用基础设施层发送消息到 RabbitMQ。
5. **基础设施层**：`HomeworkProducer` 发送消息，同时 `HomeworkRepositoryImpl` 保存聚合根到 MySQL。
6. **接口层**：返回 “处理中” 页面给学生。
7. **AI 服务**：`HomeworkConsumer` 消费消息，调用 OCR 和大模型，完成后回调 Java 接口。
8. **接口层**：`AICallbackController` 接收结果，调用应用层更新聚合根。
9. **领域层**：`HomeworkAggregate` 更新状态为 `SUCCESS`，发布 `HomeworkCorrectedEvent`。
10. **应用层**：监听到事件，更新错题本和知识点掌握情况。
11. **接口层**：通过 WebSocket 推送结果给学生（如果在线）。

## 5.2 老师端 - 作业发布与批量 AI 批改全流程

**核心需求**：老师发布班级作业→学生提交→AI 自动批量批改→自动收集全班错题与知识点掌握情况→生成班级学情报告

1. **接口层**：老师在页面填写作业信息（班级、题目列表、截止时间、评分标准），`TeacherHomeworkController` 接收请求，校验老师的班级管理权限。

2. **应用层**：`HomeworkAppService` 编排流程，通过`Assembler`将 DTO 转换为`HomeworkAggregate`（作业聚合根）。

3. **领域层**：`HomeworkAggregate` 初始化作业实体，关联题目与班级，设置作业状态为`PUBLISHED`，**发布`HomeworkPublishedEvent`（作业发布领域事件）**。

4. **基础设施层**：

    - `HomeworkRepositoryImpl` 持久化作业聚合根到 MySQL；
    - 应用层监听到`HomeworkPublishedEvent`，通过`RabbitMQ`向班级内所有学生推送作业通知，同时更新 Redis 缓存的班级作业列表。



5. **学生提交作业**：学生在截止时间前提交答案，流程同 5.1，生成`HomeworkSubmittedEvent`，作业答案状态设为`SUBMITTED`。

6. **批量批改触发**：作业截止时间到后，`xxl-job`定时任务触发批量批改，调用`HomeworkAppService`。

7. **领域层**：`HomeworkDomainService` 筛选出所有已提交的作业答案，**发布`HomeworkBatchCorrectionEvent`（批量批改事件）**。

8. **基础设施层**：

    - 消息生产者将批改任务批量发送到 RabbitMQ 的批改专用队列；
    - Python AI 服务的消费者批量消费任务，依次完成「客观题答案比对→主观题大模型评分→错误步骤提取→知识点关联」，生成批改结果。



9. **结果回调与更新**：AI 服务完成单份作业批改后，回调 Java 接口，`AICallbackController` 接收结果，调用`HomeworkAppService`。

10. **领域层**：`HomeworkAggregate` 更新批改得分、错误详情、批改状态为`CORRECTED`，**发布`HomeworkCorrectedEvent`（单份作业批改完成事件）**。

11. **后续链路编排**：

    - 应用层监听到事件，调用`LearningDomainService`，自动将错题录入学生错题本，更新对应知识点的掌握程度；
    - 所有作业批改完成后，**发布`HomeworkBatchCorrectedEvent`（全量批改完成事件）**，触发班级学情报告生成，推送给老师端。

## 5.3 老师端 - 拍题录入题库（含难认字校正）流程

**核心需求**：老师拍照录入题目→OCR 识别→难认字自动校正→知识点自动打标→入库题库

1. **接口层**：老师上传题目图片 / 截图，`QuestionController` 接收请求，校验老师的题库管理权限。

2. **应用层**：`QuestionAppService` 编排流程，先调用基础设施层将图片存入 MinIO，生成图片访问链接。

3. **领域层**：创建`QuestionAggregate`（题目聚合根），设置初始状态为`RECOGNIZING`，**发布`QuestionRecognitionEvent`（题目识别事件）**。

4. **基础设施层**：

    - 消息生产者将识别任务发送到 RabbitMQ；
    - Python AI 服务消费任务，先通过 PaddleOCR 完成版面分析与文字识别，再调用大模型完成「难认字 / 潦草字校正→公式识别→题干 / 选项 / 答案拆分」，返回结构化题目内容。



5. **结果回调与知识点打标**：AI 服务返回识别结果后，回调 Java 接口，`QuestionAppService` 调用`QuestionDomainService`。

6. **领域层**：

    - `QuestionDomainService` 调用知识图谱能力，提取题目核心知识点，关联知识点标签，完成题目结构化；
    - `QuestionAggregate` 更新题干、答案、知识点、难度等属性，状态设为`AUDITING`，**发布`QuestionRecognizedEvent`（题目识别完成事件）**。



7. **人工审核与入库**：

    - 接口层将识别结果返回给老师，老师人工校正后确认提交；
    - 应用层调用`QuestionRepositoryImpl` 将题目聚合根持久化到 MySQL，同时将题目题干与知识点向量化存入 Milvus 向量库；
    - 领域层更新题目状态为`PUBLISHED`，**发布`QuestionImportedEvent`（题目入库事件）**，更新 Redis 缓存的热门题库。

## 5.4 老师端 - AI 备课全流程

**核心需求**：老师输入备课课题→AI 生成教案框架→匹配题库例题→生成完整教案→保存与导出

1. **接口层**：老师在页面填写备课信息（学科、年级、课题、课时目标、重难点），`LessonPlanController` 接收请求。

2. **应用层**：`LessonPlanAppService` 编排流程，将备课参数封装后，调用基础设施层的 AI 服务客户端。

3. **基础设施层**：通过 HTTP 调用 Python AI 服务的备课接口，LangChain 编排大模型生成教案核心框架（教学目标、教学流程、板书设计、作业布置）。

4. **领域层**：

    - 基于生成的教案框架，`QuestionDomainService` 从题库中匹配与课题知识点匹配、难度适配的例题、课堂练习题、课后作业题；
    - 创建`LessonPlanAggregate`（教案聚合根），关联教案内容、题目列表、知识点，**发布`LessonPlanGeneratedEvent`（教案生成事件）**。



5. **接口层**：将完整教案与推荐题目返回给老师，老师通过富文本编辑器在线调整教案内容、增删题目。

6. **保存与持久化**：老师确认保存后，应用层调用`LessonPlanRepositoryImpl` 持久化教案聚合根到 MySQL；领域层更新教案状态为`OFFICIAL`，**发布`LessonPlanSavedEvent`（教案保存事件）**。

7. **扩展链路**：应用层监听到事件后，可自动将教案关联到对应班级的课程表，支持老师一键导出 PDF/Word 格式教案。

## 5.5 学生端 - 引导式 AI 答疑与薄弱点定位流程

**核心需求**：学生提问→AI 不直接给答案→逐步引导讲解→定位薄弱知识点→更新学生知识画像

1. **接口层**：学生输入问题（文字 / 图片），`StudentQAController` 接收请求，校验学生身份，初始化多轮对话上下文。

2. **应用层**：`LearningQAAppService` 编排流程，若为图片提问，先调用 OCR 服务识别题干，再封装对话参数，调用 AI 服务客户端。

3. **基础设施层**：调用 Python AI 服务的引导式答疑接口，通过 LangChain 管理对话记忆，严格遵循「不直接给答案→先提问引导学生说出解题思路→定位卡壳环节」的 Prompt 规则。

4. **多轮引导交互**：

    - AI 返回引导式问题，接口层渲染到学生端页面；
    - 学生回复自己的思路，重复上述流程，直到定位到学生的薄弱知识点环节。



5. **领域层**：

    - 定位到薄弱知识点后，`LearningDomainService` 校验知识点与题目的关联关系，**发布`StudentWeakPointLocatedEvent`（学生薄弱点定位事件）**；
    - 更新`LearningRecordAggregate`（学习记录聚合根），标记该知识点的掌握等级为`NEED_IMPROVE`，补充薄弱点原因。



6. **知识点讲解与闭环**：

    - 应用层基于定位的薄弱点，调用 AI 服务生成针对性的知识点讲解，返回给学生；
    - 讲解完成后，AI 推送 1 道基础验证题，确认学生掌握情况，领域层根据答题结果更新知识点掌握等级，**发布`KnowledgePointMasteryUpdatedEvent`（知识点掌握更新事件）**。

## 5.6 学生端 - 举一反三题型推荐流程

**核心需求**：基于当前题目 / 错题，推荐同知识点、同题型、难度梯度适配的变式题，实现巩固练习

1. **接口层**：学生在错题详情 / 题目详情页，点击「举一反三」按钮，`QuestionRecommendController` 接收请求，传入目标题目 ID 与学生 ID。

2. **应用层**：`QuestionRecommendAppService` 编排流程，先调用`QuestionRepository` 获取目标题目的知识点、题型、难度等核心属性。

3. **领域层**：

    - `QuestionDomainService` 提取推荐规则：同核心知识点、同题型、难度上下浮动 1 级、排除学生已做过的题目；
    - **发布`SimilarQuestionRecommendEvent`（相似题推荐事件）**。



4. **基础设施层**：

    - 监听到事件后，调用 Milvus 向量库，将目标题目的题干 + 知识点向量化，检索 Top10 相似题目向量；
    - 基于领域层的推荐规则，对检索结果做过滤与重排序，筛选出 3-5 道最优变式题。



5. **领域层**：`QuestionAggregate` 组装推荐题目的题干、答案、解析，**发布`QuestionRecommendedEvent`（题目推荐完成事件）**。

6. **接口层**：将推荐题目列表返回给学生，学生答题后，自动进入「批改→错题收录→知识点掌握更新」的闭环流程。

## 5.7 学生端 - 错题本自动生成与复习流程

**核心需求**：自动收录全场景错题→按知识点分类→支持错题重做→生成复习计划

1. **触发源**：作业批改完成、考试批改完成、单题拍题批改完成后，发布的`HomeworkCorrectedEvent`事件。

2. **应用层**：监听到事件后，调用`ErrorBookAppService`，提取错题 ID、学生 ID、错误原因、关联知识点。

3. **领域层**：

    - `LearningDomainService` 校验错题是否已存在于错题本，避免重复收录；
    - 创建 / 更新`ErrorBookAggregate`（错题本聚合根），关联学生、错题、错误原因、知识点、订正状态，**发布`ErrorBookRecordCreatedEvent`（错题收录事件）**。



4. **基础设施层**：`ErrorBookRepositoryImpl` 持久化错题本记录到 MySQL，更新 Redis 缓存的学生错题统计数据。

5. **错题复习流程**：

    - 学生进入错题本页面，`ErrorBookController` 接收请求，支持按知识点、错题时间、订正状态筛选；
    - 学生发起错题重做，提交答案后，进入自动批改流程，批改完成后，领域层更新错题的订正状态为`CORRECTED`，**发布`ErrorBookRevisedEvent`（错题订正完成事件）**；
    - 若订正正确，更新对应知识点的掌握等级；若订正错误，标记为高频错题，触发二次推荐相似题。



6. **复习计划生成**：基于艾宾浩斯遗忘曲线，定时任务触发，为学生生成错题复习计划，推送复习提醒。

## 5.8 学生端 - 对话情绪识别与适配流程

**核心需求**：识别学生对话中的情绪状态，适配对应的回复语气与引导策略，避免学生厌学情绪

1. **接口层**：学生在 AI 答疑对话中发送消息，`StudentQAController` 接收请求，同步将对话文本传入情绪识别流程。

2. **应用层**：`EmotionRecognitionAppService` 编排流程，调用基础设施层的 AI 情绪识别客户端。

3. **基础设施层**：Python AI 服务通过微调后的 BERT 中文情绪分类模型，对学生对话文本做分类，输出情绪标签（积极、困惑、焦虑、浮躁、抵触等）与置信度，返回给 Java 后端。

4. **领域层**：

    - `LearningDomainService` 基于情绪标签，判断学生当前的学习状态，**发布`StudentEmotionRecognizedEvent`（学生情绪识别事件）**；
    - 更新`LearningRecordAggregate`，记录学生的情绪波动，若为负面情绪，标记需要重点关注。



5. **回复适配**：

    - 应用层将情绪标签传入 AI 答疑接口，LangChain 动态调整 Prompt 的回复策略：

        - 困惑情绪：放慢引导节奏，补充基础知识点讲解；
        - 焦虑 / 浮躁情绪：先安抚情绪，拆解题目为更小的步骤，降低难度梯度；
        - 抵触情绪：暂停题目讲解，推送趣味知识点，缓解厌学情绪。





6. **预警链路**：若学生连续 3 轮对话均为负面情绪，**发布`StudentNegativeEmotionWarningEvent`（负面情绪预警事件）**，同步推送提醒给对应老师与绑定家长。

## 5.9 家长端 - 学生学情汇总报表生成流程

**核心需求**：自动生成学生学习情况汇总，包含成绩趋势、知识点掌握、错题情况、作业完成度

1. **接口层**：家长进入学情总览页面，`ParentLearningReportController` 接收请求，校验家长与学生的绑定权限，仅允许查看已审核通过的绑定学生数据。

2. **应用层**：`LearningReportAppService` 编排流程，聚合多维度数据：

    - 调用`HomeworkRepository` 获取学生近 1 个月的作业完成率、正确率、成绩趋势；
    - 调用`LearningRepository` 获取学生知识点掌握雷达图、高频薄弱知识点；
    - 调用`ErrorBookRepository` 获取学生错题总量、订正率、高频错题类型。



3. **领域层**：`LearningReportDomainService` 对聚合数据做统计分析，生成学情结论与提升建议，创建`LearningReportAggregate`（学情报告聚合根），**发布`LearningReportGeneratedEvent`（学情报告生成事件）**。

4. **基础设施层**：

    - 报表数据通过 ECharts 生成可视化图表（成绩趋势折线图、知识点掌握雷达图、错题分布饼图）；
    - 支持家长一键导出 PDF 格式的学情报告，定时任务可每月自动生成月度报告，推送给家长端。



5. **扩展链路**：若报告中出现连续下滑的成绩、多个知识点掌握不达标，**发布`StudentLearningWarningEvent`（学习预警事件）**，同步推送给对应老师。

## 5.10 家长端 - 学生账号绑定与权限审核流程

**核心需求**：家长申请绑定学生账号→老师审核→绑定成功后开通学情查看权限

1. **接口层**：家长在页面填写学生学号、学生姓名、与学生的关系，提交绑定申请，`ParentBindController` 接收请求，校验家长账号状态。

2. **应用层**：`UserBindAppService` 编排流程，校验学生账号是否存在，避免重复提交绑定申请。

3. **领域层**：

    - `UserDomainService` 创建绑定申请实体，设置状态为`PENDING`，关联家长 ID、学生 ID、对应班级老师 ID；
    - **发布`StudentBindApplyEvent`（绑定申请事件）**。



4. **基础设施层**：`UserBindRepositoryImpl` 持久化绑定申请到 MySQL，应用层监听到事件后，给对应老师推送审核通知。

5. **老师审核流程**：

    - 老师在审核页面查看绑定申请，确认家长与学生的关系真实性，选择通过 / 驳回；
    - `TeacherBindAuditController` 接收审核结果，调用`UserBindAppService`。



6. **领域层**：

    - 审核通过：更新绑定申请状态为`APPROVED`，更新家长账号的权限，关联学生 ID，**发布`StudentBindApprovedEvent`（绑定通过事件）**；
    - 审核驳回：更新绑定申请状态为`REJECTED`，填写驳回原因，**发布`StudentBindRejectedEvent`（绑定驳回事件）**。



7. **结果通知**：应用层监听到事件后，给家长推送审核结果通知，审核通过的家长自动开通对应学生的学情查看权限。
