# 错误报告流程

## 触发条件

当 subagent 发现以下问题时，必须报告：

| 问题类型 | 说明 |
|----------|------|
| 设计问题 | 设计文档与需求不符、设计不合理 |
| 接口问题 | 接口契约定义不清晰、缺失 |
| 依赖问题 | 跨领域依赖缺失、依赖顺序错误 |
| 实现问题 | 无法按设计实现、技术障碍 |
| 测试问题 | 测试无法通过、覆盖率无法达标 |

## 报告格式

```markdown
⚠️ 问题报告

**问题类型：** [设计问题/接口问题/依赖问题/实现问题/测试问题]

**问题描述：**
[清晰描述发现的问题]

**影响范围：**
[说明问题影响哪些功能或模块]

**建议方案：**
[提供可能的解决方案]

**是否需要人工介入：** [是/否]
```

## 问题类型详细说明

### 设计问题

```markdown
⚠️ 问题报告

**问题类型：** 设计问题

**问题描述：**
设计文档中用户认证流程缺少密码加密环节，存在安全风险。

**影响范围：**
用户认证功能、数据安全

**建议方案：**
1. 在设计中添加密码加密环节（推荐 BCrypt）
2. 或确认是否使用其他安全方案

**是否需要人工介入：** 是
```

### 接口问题

```markdown
⚠️ 问题报告

**问题类型：** 接口问题

**问题描述：**
设计文档中 Homework 需要调用 User 接口获取用户信息，但接口契约未定义。

**影响范围：**
作业批改功能中的用户信息获取

**建议方案：**
需要在接口契约中添加：
- 接口路径：/api/users/{id}
- 返回格式：UserResponse

**是否需要人工介入：** 是
```

### 实现问题

```markdown
⚠️ 问题报告

**问题类型：** 实现问题

**问题描述：**
按照设计文档要求，需要在 Domain 层使用 Spring Security，但这违反了 DDD 的分层原则。

**影响范围：**
Domain 层的权限控制

**建议方案：**
1. 将权限控制移至 Application 层
2. 或修改设计文档调整技术方案

**是否需要人工介入：** 是
```

## 处理流程

```
Subagent 发现问题
       │
       ▼
输出问题报告
       │
       ▼
主 Agent 接收报告
       │
       ├──→ 可自行解决 → 继续执行
       │
       └──→ 需人工介入 → 等待用户确认
```

## 原则

1. **不隐藏问题** - 发现问题必须报告
2. **不擅自修改** - 设计问题不自行修改
3. **提供方案** - 每个问题提供可行建议
4. **等待确认** - 需人工介入的问题等待确认后再继续
