# 头脑风暴：需求引导

## 概述

通过自然的协作对话，引导用户澄清需求，完成后调用 `ai-edu-architect-design` 进行设计。

<HARD-GATE>
在完成需求引导之前，不要编写任何代码或生成设计文档。
</HARD-GATE>

## 核心职责

brainstorming **只负责**：
1. 探索项目背景
2. 引导用户澄清需求
3. 提出实现方案选项
4. 识别涉及的 Bounded Context
5. 判定任务类型（A/B/C）
6. 输出需求概述
7. **调用 ai-edu-architect-design**

brainstorming **不负责**：
- ❌ 输出设计文档
- ❌ 输出 API 文档
- ❌ 输出测试文档
- ❌ 任务分配
- ❌ 子 agent 协调

## 检查清单

按顺序完成：

1. **探索项目背景** — 检查相关文件、文档、最近提交
2. **提出澄清问题** — 一次一个，理解目的/约束/成功标准
3. **提出 2-3 种方案** — 包含权衡和你的建议
4. **识别涉及的 Bounded Context**
5. **判定任务类型** — A/B/C
6. **输出需求概述**
7. **调用 ai-edu-architect-design**

## 需求引导流程

### Step 1: 探索项目背景

检查：
- 相关代码文件
- 现有设计文档
- 最近 git 提交
- 项目架构

### Step 2: 提出澄清问题

**一次一个问题**，理解：
- 目的：为什么需要这个功能？
- 约束：有什么技术或业务限制？
- 成功标准：怎么算完成？

**示例问题：**
```
- 这个功能主要用户是谁？
- 预期的使用场景是什么？
- 有没有参考的竞品或设计？
- 需要支持哪些边界情况？
```

### Step 3: 提出方案选项

提出 2-3 种实现方案，包含：
- 每种方案的优点
- 每种方案的缺点
- 你的推荐和理由

### Step 4: 识别 Bounded Context

| Context | 负责领域 | 对应 Subagent |
|---------|----------|---------------|
| **User** | 用户、权限 | `ai-edu-coder-user` |
| **Question** | 题库、知识点 | `ai-edu-coder-question` |
| **Homework** | 作业、批改 | `ai-edu-coder-homework` |
| **Learning** | 错题本、掌握度 | `ai-edu-coder-learning` |
| **Organization** | 组织架构 | `ai-edu-coder-organization` |

### Step 5: 判定任务类型

| 类型 | 条件 | 说明 |
|------|------|------|
| **A** | 单一 Context | 单领域任务 |
| **B** | 多 Context，独立 | 可并行开发 |
| **C** | 多 Context，有依赖 | 需按顺序开发 |

### Step 6: 输出需求概述

```markdown
## 需求概述

### 功能描述
[一句话描述功能]

### 用户场景
[主要使用场景]

### 涉及领域
| Context | 涉及内容 |
|---------|----------|
| User | 用户认证 |

### 任务类型
- [ ] 类型 A：单领域任务
- [ ] 类型 B：多领域独立任务
- [ ] 类型 C：多领域有依赖任务

### 依赖关系（类型 C 时填写）
```
User Context
    │
    ▼
Homework Context
```

### 方案选择
[用户确认的方案]
```

### Step 7: 调用设计 Agent

**调用 ai-edu-architect-design：**

```
请调用 ai-edu-architect-design agent

需求概述：
- 功能描述：xxx
- 涉及领域：User, Homework
- 任务类型：C
- 依赖关系：User → Homework

用户确认的方案：xxx
```

## 关键原则

- **一次一个问题** - 不要用多个问题压倒用户
- **引导而非假设** - 让用户做决定
- **只做引导** - 不输出设计文档
- **调用设计** - 完成引导后调用 ai-edu-architect-design
- **人工审核** - 设计完成后需要人工审核才能继续
